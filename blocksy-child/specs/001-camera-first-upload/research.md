# Research: Camera-First Photo Integrity

## 1. Intercepting the Upload (Backend)
**Decision**: Hook into WordPress's `wp_handle_upload_prefilter` and target requests originating from Directorist's upload mechanisms.
**Rationale**: Directorist's `/directorist/v1/temp-media-upload` REST API endpoint ultimately uses WordPress core functions (like `wp_handle_upload`) to process the file. By hooking into `wp_handle_upload_prefilter`, we catch the file right before it is moved and saved, allowing us to inspect its temporary path (`$file['tmp_name']`). We can check if `$_POST['action']` or REST API routes match the Directorist upload to isolate the validation to listing images only.
**Alternatives considered**: Using `rest_request_before_callbacks` for the specific `directorist/v1/temp-media-upload` route. However, extracting the `tmp_name` directly from the raw REST request is more complex than using the standard `wp_handle_upload_prefilter` which already gives us the `$_FILES` array in a clean format.

## 2. Validating EXIF Data for "Live Camera" Capture
**Decision**: Use PHP's built-in `exif_read_data()` on the temporary file. We will check for the presence of standard camera-specific EXIF tags, primarily `DateTimeOriginal` or `Make`/`Model`, and ensure no software tags indicating editing (e.g., `Software` like Adobe Photoshop) are present.
**Rationale**: `exif_read_data()` is native to PHP and fast. Live camera captures invariably contain `DateTimeOriginal`. Gallery or web-downloaded images often have this stripped or replaced with software tags. The fail-closed policy means if `exif_read_data()` returns false or lacks these keys, the image is rejected.
**Alternatives considered**: Using a 3rd-party EXIF library via Composer. Rejected because it violates the "no build pipeline" constraint from the project's technical context, and PHP's native function is sufficient for this check.

## 3. Verified Merchant Exemption
**Decision**: Check `get_user_meta( get_current_user_id(), 'ph_verified_merchant', true )` within the upload prefilter. If it equals `1` or `true`, bypass the EXIF validation.
**Rationale**: Directly follows the Core Principle V and VI constraints.

## 4. Frontend UI Restriction (`capture="environment"`)
**Decision**: Enqueue a custom JavaScript file (`camera-upload.js`) that attaches to the Directorist add-listing form. It will locate the file input (often rendered dynamically by Dropzone or similar in Directorist) and force `accept="image/*"` and `capture="environment"`. We will use `wp_localize_script` to pass a boolean indicating if the current user is verified. If verified, the JS will not apply the `capture` attribute.
**Rationale**: Provides immediate UI feedback and prevents the gallery picker from opening. Since Directorist might load the upload field dynamically, the JS may need to use event delegation or `MutationObserver` to ensure the attributes are applied when the input is injected into the DOM.
**Alternatives considered**: Modifying Directorist templates. This is an option, but the actual file input might be generated by a JS library (like Dropzone.js) inside Directorist, meaning a template override (`directorist/`) might not be sufficient to change the `<input type="file">` attributes. Enqueueing JS provides a robust overlay.
